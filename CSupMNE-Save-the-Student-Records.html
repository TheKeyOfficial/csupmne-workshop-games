Phones can’t reach the page? Hosting help
  • Simplest: upload this file to any web space (university server, Netlify Drop, GitHub Pages) and open it from there — then the QR works for any phone, even on mobile data.
  • Local Wi-Fi: in the file’s folder run python3 -m http.server 8080, open http://<your-laptop-ip>:8080/CSupMNE-Save-the-Student-Records-Game.html, and put the phones on the same Wi-Fi. Note: eduroam often blocks device-to-device traffic — a phone hotspot for the laptop + phones works reliably.
  • Joining phones need internet access (the connection is brokered via WebRTC).
  • Plan B is built in: if the network misbehaves, click through to the catalog and enter choices yourself — the whole game runs without phones.
The scenario

CampusOne — your student records system

THE SYSTEM
  • 12,000 student records — enrolment, grades, transcripts
  • PostgreSQL, 40 GB, on a single VM in the campus server room
  • Document share, 280 GB — scans, transcripts, exam protocols
  • Two IT staff, nightly window 22:00–06:00
  • Current setup: nightly backup to a NAS in the same rack
YOUR MISSION

Each team has 15 coins to upgrade the backup setup from a catalog of measures.

You can't afford everything — the catalog costs 38. Choose your trade-offs like a real IT team.

When all plans are locked, disaster cards hit every team equally. The simulation scores what survives: data, freshness (RPO), downtime (RTO), and proof.

The rule hiding in the game: 3-2-1-1-0

Take these home

  • Backups are the first target — in 94% of ransomware attacks, criminals go for them before you notice anything.
  • Reachable means killable — a copy on the same domain with the same credentials is not a safety net.
  • Retention saves you from yourself — slow-burn mistakes need versions, not just last night's copy.
  • A backup is a hypothesis; a restore is the proof — the teams that tested recovered fastest.
